# M4 ‚Äî Design Decisions

## M4-01 ‚Äî Iteration log schema

**Decision:** `iteration-log.json` contains:

- `session_id` ‚Äî session identifier
- `iteration` ‚Äî iteration number (integer)
- `strategy` ‚Äî what approach the worker chose
- `decision_rationale` ‚Äî why this approach was chosen
- `next_action` ‚Äî what the worker intends to do next

**Discarded fields:** `quality_score` (no real consumer in the system ‚Äî L2/L3/L4 evaluate
quality independently), `alternatives_considered` (high overhead, low signal ‚Äî see M0-03).

**Rationale:** Three fields give the next iteration enough context without overhead.

---

## M4-02 ‚Äî Iteration log location

**Decision:** `artifacts/bdralph/iteration-log.json` ‚Äî single file, overwritten after
each iteration.

**Rationale:** The worker needs context from the previous iteration, not the full history.
L4 traces (M3) already cover historical context. A single file is simpler and consistent
with M0-02.

---

## M4-03 ‚Äî Iteration log cleanup

**Decision:** `artifacts/bdralph/iteration-log.json` is deleted at the start of each
session, alongside traces and other session state.

**Rationale:** Prevents context bleed between unrelated tasks. A previous session's
`iteration-log.json` contains strategy and rationale for a different task ‚Äî reading it
in a new session would mislead the worker.

**Condition to reopen:** If M6b implements explicit session continuation (resuming a
BLOCKED task), revisit this policy. Persistence between sessions is a feature that
requires CLI support to distinguish "new task" from "continue previous task". Do not
implement persistence without that distinction.

---

## M4-04 ‚Äî Who writes the iteration log

**Decision:** The worker writes `iteration-log.json` at the end of each iteration,
before the review pipeline runs.

**Rationale:** The log captures the worker's own assessment of its strategy and next
action ‚Äî it must be written by the worker, not inferred by the loop.

---

## M4-05 ‚Äî Who reads the iteration log

**Decision:** The worker reads `iteration-log.json` at the start of each iteration
(if it exists). The loop passes the file path in the worker prompt but does not
interpret its contents.

**Rationale:** The loop is an orchestrator, not a reasoner. Interpretation of the
iteration log belongs to the worker.
