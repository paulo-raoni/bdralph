# decisions/M0.md — Planning Session Decisions

Full detail for all decisions recorded in `DECISIONS.md` under M0.

---

## M0-01 — Trace system: vertical hierarchy within iteration

**Decision:** Traces within the current iteration flow vertically:

```
Worker works
    ↓
L1  → writes artifacts/bdralph/traces/l1-iteration-N.json  (bash, $0)
    ↓
L2  → reads L1 → writes traces/l2-iteration-N.json
    ↓
L3  → reads L1+L2 → writes traces/l3-iteration-N.json
    ↓
L4  → reads L1+L2+L3 → writes traces/l4-iteration-N.json
    ↓
SHIP or REVISE
```

**What the worker reads before each iteration:**
- Its own `iteration-log.json` from all previous iterations
- L4 traces from the last N previous iterations (N configurable, default 3)
- Worker does NOT read current iteration traces — they don't exist yet when it starts
- L1–L4 do NOT read past iterations — they are evaluators of the present moment

**Access is optional and additive** — if traces don't exist, each layer operates as today. Never blocking.

**Rationale:** Worker needs historical context to avoid repeating failed approaches. Layers need only the current state to evaluate fairly.

## M0-02 — iteration-log approved fields

**Decision:** `iteration-log.json` contains:
- `strategy` — what approach the worker chose
- `decision_rationale` — why this approach was chosen
- `next_action` — what the worker intends to do next

**Rationale:** These three fields give the next iteration enough context without overhead.

---

## M0-03 — iteration-log discarded fields

**Discarded:** `quality_score`, `alternatives_considered`

**Reasons:**
- `quality_score`: no real consumer in the system. L2/L3/L4 already evaluate quality independently.
- `alternatives_considered`: high overhead to produce, low signal for the next iteration. Worker would spend tokens listing alternatives it won't pursue.

**Condition to reopen `quality_score`:** if observability tooling is built that consumes and visualizes it (M6+).
**Condition to reopen `alternatives_considered`:** if a specific case shows the worker repeating the same bad alternative and this field would have prevented it.

---

## M0-04 — Bidirectional interface: operator-signal.json

**Decision:** `artifacts/bdralph/operator-signal.json` is the operator-to-loop communication channel.

**Verified:** at the start of each iteration, not during.

**Formats:**
```json
{"action": "stop-now"}
{"action": "stop-after-this"}
{"action": "stop-on-fail"}
{"action": "message", "content": "..."}
```

**Rationale:** Checking once per iteration is sufficient and avoids complexity of mid-iteration interruption.

## M0-05 — Shutdown: 5 statuses, heartbeat, atomic write

**Shutdown statuses:**

| Status | Cause | Report |
|---|---|---|
| `completed` | Normal SHIP | ✅ complete |
| `stopped-now` | Operator signal | ✅ partial |
| `stopped-after` | Operator signal | ✅ complete |
| `interrupted` | SIGINT / SIGTERM / SIGHUP | ✅ partial via signal handler |
| `crashed` | kill -9 / power loss | ✅ partial via last heartbeat |

**Heartbeat:** file updated every 5s, 30s threshold to declare crash.

**Atomic write:** always write to `.tmp`, rename to final name. Never corrupted by shutdown.

## M0-06 — CLI wrapper

**Decision:** bdralph is the operator's terminal. Claude Code runs headless via `claude -p --dangerously-skip-permissions` — invisible to the operator.

**Flags preserved from the factory loop:**

| Flag | Default | Description |
|---|---|---|
| `--max N` | 10 | Maximum iterations |
| `--budget USD` | 0.50 | Session budget |
| `--worker sonnet\|opus\|auto` | sonnet | Worker model |
| `--escalate-after N` | 3 | Consecutive REVISEs before auto-escalation |
| `--reviewer-mode pipeline\|single` | pipeline | Review pipeline mode |

**`BDRALPH_NO_UI=1`** disables the panel for CI and headless environments.

## M0-07 — Panel layout: fixed order, 3 breakpoints

**Fixed order (immutable):** alerts (max visual priority) → Second Mind → loop

**Breakpoints:**

Medium terminal (80–119 columns):
┌──────────────────────────────┐

├───────────────┬──────────────┤
│  LOOP         │  SECOND MIND │

Narrow / stacked:
┌──────────────────────────────┐
│  ⚠️ ALERTS                  │
├──────────────────────────────┤
│  SECOND MIND                 │

**Implementation stages:**
- M2 — basic loop, fixed layout, real-time streaming
- M6 — Second Mind + alerts

## M0-08 — Second Mind: on-demand, 3 triggers

**Decision:** Second Mind is on-demand, not an active sentinel. Continuous Awareness Mode was discarded (see M0-11).

**3 activation triggers:**
1. Explicit by operator
2. Iteration threshold without SHIP
3. L4 signal after N consecutive REVISEs on the same point

**Interface:** `bdralph ask "question"` in a separate terminal → writes to `operator-signal.json` → panel shows response.

**Gas Town:** Second Mind has global visibility. Worker remains completely isolated.

---

## M0-09 — Gas Town: explicit opt-in

**Decision:** `BDRALPH_GAS_TOWN=enabled` required. Off by default.

**Flow:** external copilot (claude.ai) declares dependencies in parameterized JSON → Second Mind validates → loops fire in isolated worktrees → deterministic bash watchdog monitors → integration ralph at the end → Second Mind does coherence check.

**Watchdog triggers:** two loops touching the same file, crash, or N iterations without progress.

**Budget:** divided proportionally between loops at start — prevents one loop from consuming everything.

---

## M0-10 — Agnostic mode: executor CLI

**Decision:** Agnostic mode means multiple agent CLIs, NOT LLM via API directly.

**LLM via API directly discarded:** no real agency in the filesystem — doesn't have the tools needed to act as a worker in this system.

**Claude Code:** primary and default executor.
**Codex:** dedicated experimental milestone, no date.
**Antigravity:** agnostic execution environment from the factory — may be the runtime for bdralph's agnostic mode. Explore when reaching relevant milestones.

---

## M0-11 — Continuous Awareness Mode: discarded

**Decision:** Discarded.

**Reason:** constant cost without proportional benefit. Second Mind activated on-demand (M0-08) covers the cases that matter without the overhead of continuous monitoring.

**Condition to reopen:** if a real case shows the operator missing a critical issue that continuous monitoring would have caught, with evidence that on-demand wasn't sufficient.

## M0-14 — Provider chain env vars: BDRALPH_ prefix

**Decision:** All bdralph-specific env vars use `BDRALPH_` prefix. The factory used `RALPH_` for its chain vars (`RALPH_L2_PROVIDER_CHAIN`, etc.) — bdralph uses `BDRALPH_L2_PROVIDER_CHAIN`, `BDRALPH_L3_PROVIDER_CHAIN`, `BDRALPH_PROVIDER_FAILOVER`.

**Rationale:** clear namespace separation, no ambiguity with factory vars if both are in scope.

## M0-15 — Branch naming convention

**Decision:**

| Prefix | When to use |
|---|---|
| `feat/mN-description` | milestone implementation |
| `fix/description` | bug fix |
| `chore/description` | infra, lockfile, setup |
| `doc/description` | documentation only |
| `bootstrap/description` | initial structure |

**Origin:** pattern observed in factory handoffs (M13–M26). Not formally documented in factory's CLAUDE.md — formalizing here from the start.

**Rationale:** consistent, scannable, makes branch purpose obvious in git log and PR list.

## M0-16 — Governance files location: docs/

**Decision:** PROGRESS.md, BACKLOG.md, DECISIONS.md, decisions/, and HANDOFF_TEMPLATE.md live in docs/. CLAUDE.md and README.md stay at the repository root.

**Rationale:** docs/ is the GitHub convention for project documentation, recognizable to external contributors without implying internal AI tooling. CLAUDE.md must stay at root because Claude Code reads it from there by convention. README.md stays at root because GitHub renders it automatically.

## M0-17 — Executor invocation contract

**Decision:** Each executor has a defined invocation mode for non-interactive implementation runs:
- Claude Code CLI: claude --dangerously-skip-permissions < prompt.md
- Codex CLI: codex exec --dangerously-bypass-approvals-and-sandbox < prompt.md
- Antigravity: Terminal Policy set to "Always proceed" + Agent mode "Agent-driven" in Settings

**Rationale:** Without these modes, executors pause at every file write and bash command, requiring manual approval throughout the run. Inside the devcontainer (or Antigravity's isolated workspace), this is safe. Documented here so every prompt doesn't need to repeat the invocation instructions.
