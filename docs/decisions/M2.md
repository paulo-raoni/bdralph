# M2 — Design Decisions

## M2-01 — Streaming transport: log file + tail

**Decision:** Worker writes stdout to `UI_WORKER_OUTPUT_FILE` (already wired in ralph-loop.sh).
The Ink panel reads the file on a 200ms interval using `fs.readFileSync`.

**Alternatives discarded:**
- FIFO (named pipe): blocks the writer if the panel crashes or stops reading
- Socket (requires socat or a custom IPC mechanism): over-engineered for the use case

**Rationale:** The log file is auditable after the session in `artifacts/bdralph/`, carries no
blocking risk, and is consistent with the atomic write pattern already used for state files.

---

## M2-02 — Panel fields in M2

**Decision:** The Ink panel displays: iteration/max, worker model, accumulated cost + budget
remaining, elapsed time, and the last N lines of worker stdout.

**Rationale:** These fields are sufficient for the operator to understand loop state at a
glance. Second Mind integration and Alerts are deferred to M6.

---

## M2-03 — BDRALPH_NO_UI=1 behavior

**Decision:** When `BDRALPH_NO_UI=1` is set, the Ink panel does not start. Worker stdout goes
directly to the terminal. The session summary is still printed at the end.

**Rationale:** CI-friendly — stdout is capturable via pipe/grep. Consistent with M1b-04
(session termination outputs summary to stdout).

---

## M2-04 — Streaming line count: N=10 fixed

**Decision:** The panel displays the last 10 lines of worker output. This is fixed in M2.
A `TODO` comment marks the site for M6 dynamic height based on terminal rows.

**Rationale:** Simple and sufficient for M2. Dynamic height per breakpoint belongs to M6
(M0-07 already defines the 3 breakpoints).
