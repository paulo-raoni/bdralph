# M5 — Design Decisions

## M5-01 — L2 worker outcome classification

**Decision:** L2 receives in its prompt: the git diff, the L1 trace, and the last N lines
of worker stdout. L2 classifies the worker outcome as one of three values, recorded in
the L2 trace as `worker_outcome_classification`:

- `pass` — worker completed the task successfully
- `failure` — worker failed due to a real error (bug, broken tests, logic error)
- `safety_impediment` — worker stopped due to a legitimate constraint (sensitive path
  blocked by L1, worker recognized it cannot proceed without violating a rule)

**Rationale:** The distinction between `failure` and `safety_impediment` is critical for
SHIP-ON-FAILURE — the feature must never trigger on legitimate safety constraints, only
on real failures.

---

## M5-02 — L2 context for classification

**Decision:** L2 receives: git diff + L1 trace + last N lines of worker stdout (same N
as `BDRALPH_TRACE_HISTORY`, default 3).

**Rationale:** The L1 trace tells L2 whether sensitive paths were involved. The worker
output tail gives L2 the worker's own account of what happened. Together these are
sufficient for classification without requiring the full worker output.

---

## M5-03 — SHIP-ON-FAILURE configuration

**Decision:** SHIP-ON-FAILURE is configured in `.bdralph.config.json` at the repo root.
It is opt-in — disabled by default.

Example configuration:

```json
{
  "ship_on_failure": {
    "enabled": true,
    "triggers": [
      "all tests pass",
      "no regression introduced"
    ]
  }
}
```

**Rationale:** SHIP-ON-FAILURE is a deliberate decision to accept partial results. It must
be declared explicitly in the repo — not inferred, not enabled by default. The config file
lives at the repo root so it applies to the project, not to a specific operator session.

---

## M5-04 — Trigger list is semantic, not keyword-based

**Decision:** `triggers` is a list of natural language conditions evaluated by L2. L2
determines whether the conditions are satisfied based on the diff, L1 trace, and worker
output — not by string matching.

**Rationale:** Keyword matching produces false positives (e.g. "all tests pass" matching
a comment). Semantic evaluation via L2 is more reliable and handles negations and
paraphrases correctly.

---

## M5-05 — Behavior when .bdralph.config.json is absent

**Decision:** Absence of `.bdralph.config.json` is equivalent to `enabled: false`.
The loop proceeds normally without error or warning.

**Rationale:** Most projects will not use SHIP-ON-FAILURE. Requiring the file would add
friction for the common case. Silence is the correct behavior when a feature is not
configured.

---

## M5-06 — SHIP-ON-FAILURE trigger conditions

**Decision:** SHIP-ON-FAILURE only fires when ALL of the following are true:

1. `.bdralph.config.json` exists and `ship_on_failure.enabled` is `true`
2. L2 classifies the worker outcome as `failure` (not `safety_impediment`, not `pass`)
3. L2 evaluates the configured triggers as satisfied

If any condition is false, the loop follows the normal flow (REVISE or BLOCKED).

**Rationale:** Multiple independent conditions prevent accidental SHIPs. A
`safety_impediment` must never result in SHIP — the operator needs to investigate.
