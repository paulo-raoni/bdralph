# M6 — Design Decisions

## M6-01 — Stop modes

**Decision:** `bdralph stop` is a subcommand that writes to `operator-signal.json`.
Three flags:

- `--now` — stop immediately, in the current iteration
- `--after-this` — finish the current iteration, then stop
- `--on-fail` — stop if the next iteration fails

These map directly to the `operator-signal.json` actions defined in M0-04
(`stop-now`, `stop-after-this`, `stop-on-fail`). M6 makes them visible in the
panel and actionable via CLI without requiring the operator to write JSON manually.

**Rationale:** The signal formats were already designed in M0-04. M6 is the UX layer
on top of that contract.

## M6-02 — Second Mind activation triggers

**Decision:** Three independent triggers activate the Second Mind:

1. **Explicit** — operator runs `bdralph ask "question"` in a separate terminal.
   Writes to `operator-signal.json` with `action: "message"`.
2. **Iteration threshold** — loop reaches N iterations without SHIP, where N defaults
   to `floor(--max / 2)`. Configurable via `BDRALPH_SM_THRESHOLD`.
3. **L4 signal** — L4 emits REVISE N consecutive times on the same point, where N
   is the same as `--escalate-after` threshold.

Triggers 2 and 3 are automatic — no operator intervention required.

**Rationale:** The three triggers cover the three cases where Second Mind adds value:
operator-initiated inquiry, loop stuck without progress, and reviewer stuck on the
same issue.

---

## M6-03 — bdralph ask is a provisional interface

**Decision:** `bdralph ask "question"` is the M6 interface for explicit Second Mind
activation. It is explicitly provisional — M6b (REPL terminal) replaces it with
direct interaction inside the persistent terminal, without requiring a separate
command in another terminal.

**Rationale:** M6b requires Second Mind to be meaningful (the REPL without Second Mind
is incomplete — see M1b-05). M6 delivers the functional interface; M6b delivers the
ergonomic one.

---

## M6-04 — Second Mind context

**Decision:** Second Mind has access to the complete session context:
- All L1–L4 traces from the current session
- `iteration-log.json` (current iteration)
- Last N lines of worker output (`BDRALPH_TRACE_HISTORY` controls N)
- The operator's question (if trigger was explicit)

Worker remains completely isolated — Second Mind cannot write to files the worker reads.

**Rationale:** Without session context, Second Mind is a generic LLM with no insight
into what the loop is doing. Full context is what makes it useful. Isolation from the
worker preserves the loop's governance contract.

---

## M6-05 — Second Mind response

**Decision:** Second Mind writes its response to
`artifacts/bdralph/second-mind-response.txt`. The Ink panel displays it in the
Second Mind section. The file is overwritten on each activation.

**Rationale:** Simple, consistent with the state file pattern already used throughout
the system.

---

## M6-06 — Complete panel layout

**Decision:** M6 delivers the complete 3-section panel layout defined in M0-07:
alerts → Second Mind → loop, with all 3 breakpoints fully implemented (wide 120+,
medium 80–119, narrow/stacked, minimalist <15 lines).

M2 delivered the basic loop section with fixed layout. M6 adds the Second Mind and
alerts sections and implements full responsiveness.

**Rationale:** Completing the panel layout in M6 aligns with Second Mind delivery —
the panel is only complete when all three sections have content.
