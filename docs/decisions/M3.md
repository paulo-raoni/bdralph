# M3 — Design Decisions

## M3-01 — Trace schema

**Decision:** All layers write a JSON trace file after execution. Fields common to all layers:

- `session_id` — session identifier (format: `"20240325T104512-1234"`)
- `iteration` — iteration number (integer)
- `layer` — layer identifier (`"l1"`, `"l2"`, `"l3"`, `"l4"`)
- `timestamp_start` — ISO 8601 UTC
- `timestamp_end` — ISO 8601 UTC
- `duration_ms` — integer
- `result` — `"PASS"` | `"REVISE"` | `"SHIP"` | `"BLOCKED"`
- `cost_usd` — float (0 for L1)
- `tokens` — object with `input` and `output` integer fields (zero for L1)
- `provider` — provider string (null for L1)
- `feedback` — string

**L1 additional fields** (no LLM — cost_usd, tokens, provider are zero/null):

- `sensitive_paths_matched` — array of matched paths (empty if none)
- `files_checked` — integer
- `escalated_to_l4` — boolean

**L4 additional fields:**

- `triggered_by` — array: `["l1_escalation"]`, `["consecutive_revises"]`, or `["l1_escalation", "consecutive_revises"]`
- `consecutive_revises_at_trigger` — integer: counter value at moment L4 was called
- `l1_escalated` — boolean

`triggered_by` is an array to support both triggers occurring simultaneously.
`consecutive_revises_at_trigger` records the counter value regardless of which trigger fired.
L4 is triggered by L1 escalation (sensitive path detected in current iteration) or by
consecutive L3 REVISEs reaching the `--escalate-after` threshold. L2 does not trigger L4.

**Rationale:** Common schema enables uniform tooling across layers. Layer-specific fields
capture the information most relevant to each layer's role without bloating the common schema.

---

## M3-02 — Trace file location

**Decision:** `artifacts/bdralph/traces/` — flat directory, one file per layer per iteration.

**Rationale:** Flat structure simplifies glob operations. The worker reads L4 traces from
the last N iterations via `traces/l4-iteration-*.json | sort | tail -N`. With a default
max of 10 iterations, the directory never grows large enough to be a problem.
Hierarchical structure (`iteration-N/lN.json`) would require iterating directories for
cross-iteration reads.

---

## M3-03 — Trace file naming

**Decision:** `lN-iteration-N.json` — e.g. `l1-iteration-3.json`, `l4-iteration-10.json`.

**Rationale:** Descriptive, no abbreviations, unambiguous. Layer and iteration number are
immediately readable without parsing.

---

## M3-04 — Trace cleanup

**Decision:** `artifacts/bdralph/traces/` is deleted at the start of each session
(before the first iteration), alongside `work-summary.txt` and `work-complete.txt`.

**Rationale:** Consistent with existing session state cleanup. Post-session observability
(drill-down into historical traces) is deferred to M6+.

**Condition to reopen:** If M6b implements explicit session continuation, revisit this
policy alongside M4-03.

---

## M3-05 — N configurability (worker trace history)

**Decision:** The number of L4 traces the worker reads before each iteration is controlled
by `BDRALPH_TRACE_HISTORY` env var, default 3.

**Rationale:** Not a CLI flag — this is a tuning parameter for advanced users, not
something that belongs in `--help`. Env vars are already the pattern for provider chain
configuration (`BDRALPH_L2_PROVIDER_CHAIN`, etc.).
